## C# Coding Conventions

### Naming
- Use **PascalCase** for classes, records, structs, enums, methods, properties, and events.
- Use **camelCase** for local variables and method parameters.
- Add here to the guidelines as specified by Microsoft: https://learn.microsoft.com/en-us/dotnet/standard/design-guidelines/capitalization-conventions
- All Task returning methods should end with `Async`, e.g., `GetUserAsync`.
- Prefix private fields with an underscore (`_`), e.g., `_myField`.
- Prefix interfaces with `I`, e.g., `IService`.
- Avoid abbreviations and "magic" strings or numbers. Use constants or enums instead.

### Formatting
- Use **Allman style** braces (each brace on a new line).
- Indent with **2 spaces**, not tabs.
- Keep lines concise, ideally under 120 characters.
- Add a single blank line between method and property definitions.
- `using` directives should be placed outside the namespace declaration and sorted alphabetically, with `System.*` directives first.
- Put each class, record, enum or struct in its own file named after the type. Don't combine multiple types in a single file.

### Language Usage
- **`var` vs. Explicit Types:** Try to avoid `var`, use it only for anonymous types.
- **Avoid `dynamic`:** Try to avoid `dynamic`, use it only if anything else is impossible.
- **Asynchronous Programming:** Always use `async Task` instead of `async void`. Use `ConfigureAwait(false)` in library code where the context is not required.
- **LINQ:** Use LINQ for collection manipulation where it enhances readability. Prefer method syntax over query syntax for simplicity, unless query syntax is significantly more readable. Use extension methods like `Select`, `Where`, and `FirstOrDefault` for clarity, don't use Query syntax.
- **Null-Coalescing:** Use the `??` and `??=` operators to handle nulls concisely.
- **Exception Handling:** Catch specific exceptions rather than `System.Exception`. Only catch exceptions you can handle.
- **Immutability:** Favor immutable data structures where possible. Use `records` for data transfer objects (DTOs) and other immutable types.
- **Syntax:** Use modern languages features to avoid warnings and suggestions like:
  - Use collection expression
  - Use new keyword without type (redundant type specification)

# Git Guidelines

## Worktrees
- Use **git worktrees** for parallel feature work and isolation from the main workspace.
- Store all worktrees in a `.worktrees/` directory at the repository root.
- The `.worktrees/` directory must be in `.gitignore`.
- Name worktrees after the branch: `.worktrees/<branch-name>`.
- Create worktrees with: `git worktree add .worktrees/<branch-name> -b <branch-name>`
- Remove finished worktrees with: `git worktree remove .worktrees/<branch-name>`
- Run `git worktree prune` periodically to clean up stale references.

## Branching
- Use short, descriptive branch names in **kebab-case**: `feature/add-user-auth`, `fix/null-pointer-crash`.
- Keep branches short-lived. Merge or rebase frequently to avoid drift.
- Delete branches after merging.

## Commits
- Write concise commit messages that explain **why**, not what.
- Use conventional commit prefixes: `feat:`, `fix:`, `refactor:`, `docs:`, `test:`, `chore:`.
- Keep commits atomic — one logical change per commit.
- Do not commit secrets, credentials, or environment files.
